# 学习日志防丢失流程

## 目标与存储模型

学习日志必须把“事实记录”和“展示页面”分开：

- `<日志目录>/events/*.json` 是唯一事实源；每次学习记录创建一个新文件，既不搜索替换，也不改旧文件。
- `<日志目录>/study-log.md` 是自动生成的汇总视图。它损坏或过期时可由 `events/` 完整重建。
- `<日志目录>/backups/` 保存最近 20 个汇总页版本；备份不是事实源。

不要把日志放在临时沙箱、下载后会被清理的目录、skill 安装目录或仅存在于模型上下文中的“文件”。优先使用
用户明确指定的本地文件夹、受版本控制的私人工作区，或具有真实读写能力的云盘连接器。

## 在 Codex App / CLI / IDE 中

Codex 支持带 `scripts/` 和 `references/` 的 skill，但必须区分两个位置：

- **skill 目录**：如 `~/.agents/skills/cpa-tutor/`，只用于读取 `SKILL.md`、参考资料和脚本。
- **日志目录**：用户打开的持久工作区中的 `cpa-study-log/`，用于保存真实学习数据。

从当前被选中的 `SKILL.md` 所在目录解析 `scripts/study-log.js` 的绝对路径，再把用户工作区内的日志目录作为
`--dir` 传入。不要用当前工作目录拼出 `scripts/study-log.js`，也不要把数据写回 skill 安装目录。路径含空格或
中文时给脚本路径和日志路径加引号。

推荐让用户先在 Codex 中打开一个长期保存的 CPA 学习文件夹，然后这样提出要求：

```text
使用 $cpa-tutor。把学习日志保存在当前工作区的 cpa-study-log 目录；以后我说“记日志”时，
请用追加式日志脚本写入，并在每次写入后运行 verify。
```

若当前工作区明确是持久且可写的，用户没有另给路径时可使用 `<当前工作区>/cpa-study-log`；否则先确认目录。
Codex 沙箱拒绝写入时，只请求该日志目录所需的最小权限。不要改写到临时目录来绕过审批。云端任务或临时
worktree 中的文件不自动等于长期保存；只有提交到用户仓库或同步到外部持久存储后才算持久日志。

## 有本地命令行与持久文件权限

脚本不默认路径，必须显式传入用户的持久日志目录：

```bash
node scripts/study-log.js status --dir "<持久目录>/cpa-study-log"
node scripts/study-log.js init --dir "<持久目录>/cpa-study-log"
node scripts/study-log.js append --dir "<持久目录>/cpa-study-log" --subject "经济法" --topic "票据法律制度" --summary "完成第九章第一单元练习" --result "20 题，错 3 题" --weak-points "提示付款期限、票据抗辩" --next-step "明天重做错题"
node scripts/study-log.js verify --dir "<持久目录>/cpa-study-log"
```

处理顺序：

1. 首次使用先确认目录是持久目录，再运行 `status`。目录为空才运行 `init`。
2. 若目录里已有旧版单文件 `study-log.md`、但没有 `events/`，不要初始化或覆盖。先运行：
   `node scripts/study-log.js import-markdown --dir "<日志目录>" --file "<旧日志.md>" --subject "历史日志"`。
3. 每次只用 `append` 新增一条记录。详细内容较长时可加 `--details "..."`；不要直接编辑生成的汇总页。
4. 每次写入后运行 `verify`。脚本会校验事件数量和汇总页摘要；失败就停止继续写。
5. 汇总页丢失或被误改时运行 `rebuild --dir "<日志目录>"`，从所有事件重建。

脚本遇到损坏事件、遗留单文件或锁冲突会拒绝继续，目的是避免用空文件/旧副本覆盖有效历史。不要绕过检查；
先备份整个日志目录，再查明异常文件。

## 只有 Artifact / 项目知识库 / 普通聊天

- Artifact：先查看版本选择器和历史版本；更新应产生新版本。不要把 Artifact 当作唯一长期备份，定期下载到
  用户自己的文件夹或云盘。
- 项目知识库：它主要给模型提供检索上下文，不等于一个可安全追加的数据库。若平台没有明确的文件更新能力，
  生成新的日志文件让用户重新上传，而不是声称已修改原文件。
- 普通聊天：可以给出“本次新增记录”或完整重建文本，但必须明确尚未写入持久存储，并请用户导出。

## 发生异常时

看到“找不到匹配文本”只代表定位条件不满足，不等于文件已被系统重置。依次检查：

1. 当前打开的是不是正确路径、正确 Artifact、正确版本；是否误读了同名文件。
2. `events/` 是否仍在；若在，运行 `verify`，再运行 `rebuild`。
3. 查看 `backups/`、Artifact 版本历史、操作系统/云盘版本历史和 Git 历史。
4. 只有确认持久存储中的历史确实不存在后，才根据聊天记录重建；把重建内容标记为“恢复记录”，不要伪装成
   原始日志。
5. 恢复后再次运行 `verify`，并让用户保存一份外部备份。

绝不在不确定时运行初始化、整文件重写、删除旧日志或创建一个同名空日志。
